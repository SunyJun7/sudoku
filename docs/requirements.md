## 프로젝트 규모 판단

**규모: Small (소규모 개인 프로젝트)**

| 항목 | 내용 |
|------|------|
| 개발 인원 | 1명 (개인 프로젝트) |
| 예상 개발 기간 | 3~4주 |
| 타겟 사용자 | 2명 (부모님) |
| 배포 방식 | APK 직접 설치 (사이드로딩) |

규모 판단 근거: 단일 개발자, 외부 서비스 연동 없음, 제한적 사용자, 명확하게 정의된 기능 범위. 단, Play Store 출시를 고려한 확장 가능한 아키텍처를 초기부터 적용한다.

## 프로젝트 개요

부모님(어머니, 아버지) 전용 스도쿠 앱. Android/iOS 스마트폰을 사용하는 시니어 사용자를 대상으로 하며, 큰 글씨와 단순한 조작을 최우선으로 한다. 현재 단계는 APK 직접 설치이며, 추후 Play Store 출시를 고려하여 난이도 시스템, 퍼즐 데이터 분리, 다국어 구조를 초기 설계에 반영한다.

## 기능 요구사항

### FR-001 퍼즐 생성

- 사용자로서, 난이도(쉬움/보통/어려움)를 선택하여 새 퍼즐을 시작하고 싶다. 왜냐하면 그날의 컨디션과 실력에 맞게 즐기고 싶기 때문이다.
- 수용 기준:
  - Easy / Normal / Hard 3단계 난이도 선택 화면이 존재한다.
  - 각 난이도별로 사전 생성된 퍼즐 데이터를 로컬에 보유한다.
  - 퍼즐 데이터는 로직 코드와 분리된 데이터 레이어에 위치한다 (추후 서버 연동 가능한 구조).
  - 동일 난이도에서 매번 다른 퍼즐이 출제된다 (랜덤 선택).

### FR-002 숫자 입력

- 사용자로서, 빈 칸을 탭하고 숫자 버튼을 눌러 숫자를 입력하고 싶다. 왜냐하면 드래그 없이 탭만으로 조작하고 싶기 때문이다.
- 수용 기준:
  - 빈 칸 탭 시 해당 칸이 선택(하이라이트)된다.
  - 화면 하단에 1~9 숫자 버튼이 크게 표시된다.
  - 숫자 버튼 탭 시 선택된 칸에 숫자가 입력된다.
  - 이미 채워진 초기 숫자(힌트)는 선택/변경 불가하다.
  - 드래그, 스와이프 조작은 일절 요구하지 않는다.

### FR-003 입력 취소 (지우기)

- 사용자로서, 입력한 숫자를 지우고 싶다. 왜냐하면 실수로 잘못 입력했을 때 되돌리고 싶기 때문이다.
- 수용 기준:
  - 숫자 버튼과 동일한 크기의 "지우기" 버튼이 제공된다.
  - 지우기 버튼 탭 시 선택된 칸의 사용자 입력값이 삭제된다.

### FR-004 오류 안내

- 사용자로서, 틀린 숫자를 입력했을 때 즉시 알고 싶다. 왜냐하면 모르는 채로 진행하면 나중에 고치기 어렵기 때문이다.
- 수용 기준:
  - 입력한 숫자가 정답과 다를 경우 해당 칸이 빨간색(또는 명확히 구분되는 색)으로 표시된다.
  - 오류 표시는 칸 색상 변경으로 처리하며, 팝업/진동 등 강제 중단 인터랙션은 사용하지 않는다.
  - 오류 여부를 즉시 판단하는 방식(실시간 검증)을 기본으로 한다.

### FR-005 게임 완료 처리

- 사용자로서, 퍼즐을 완성했을 때 완료 메시지를 보고 싶다. 왜냐하면 성취감을 느끼고 다음 단계를 선택하고 싶기 때문이다.
- 수용 기준:
  - 모든 칸이 정답으로 채워지면 자동으로 클리어 화면이 표시된다.
  - 클리어 화면에는 "다시 하기", "새 게임" 버튼이 제공된다.
  - 클리어 메시지는 친근하고 긍정적인 문구를 사용한다.

### FR-006 게임 재시작 / 새 게임

- 사용자로서, 현재 퍼즐을 초기화하거나 새 퍼즐을 시작하고 싶다. 왜냐하면 막혔을 때 다시 시작하고 싶기 때문이다.
- 수용 기준:
  - 게임 화면에 "다시 시작" 버튼이 제공된다.
  - "다시 시작" 탭 시 사용자 입력값만 초기화되고 힌트 숫자는 유지된다.
  - "새 게임" 버튼은 난이도 선택 화면으로 이동한다.

### FR-007 게임 진행 상태 저장

- 사용자로서, 앱을 닫았다가 다시 열었을 때 진행 중이던 게임을 이어서 하고 싶다. 왜냐하면 전화를 받거나 잠시 앱을 닫아도 처음부터 다시 하고 싶지 않기 때문이다.
- 수용 기준:
  - 앱 종료/백그라운드 전환 시 현재 게임 상태(퍼즐 데이터, 사용자 입력값, 선택 난이도)를 로컬에 자동 저장한다.
  - 앱 재실행 시 저장된 게임이 있으면 이어서 진행한다.
  - 저장 수단은 SharedPreferences (로컬, 오프라인)를 사용한다.
  - 새 게임 시작 시 저장된 상태를 초기화한다.

### FR-008 다국어 구조 (한국어 우선)

- 사용자로서, 앱의 모든 텍스트가 한국어로 표시되어야 한다. 왜냐하면 영어에 익숙하지 않기 때문이다.
- 수용 기준:
  - 앱 내 모든 UI 텍스트는 코드에 하드코딩하지 않고 별도 문자열 리소스 파일로 관리한다.
  - 기본 언어는 한국어이다.
  - 문자열 리소스 구조는 추후 다국어 확장이 가능한 형태이다 (예: `l10n` 디렉토리 분리).

## 비기능 요구사항

### NFR-001 가독성 (시니어 친화 UI)

- 본문 폰트 크기는 최소 18sp 이상, 핵심 버튼/숫자는 24sp 이상으로 설정한다.
- 스도쿠 그리드의 각 셀 크기는 손가락으로 탭하기 충분한 크기(최소 44dp 이상)로 설정한다.
- 색상 대비는 WCAG AA 기준(명암비 4.5:1 이상)을 준수한다.

### NFR-002 조작 단순성

- 모든 핵심 기능은 탭(단순 터치) 하나로 동작해야 한다.
- 드래그, 멀티터치, 길게 누르기(long press) 조작을 핵심 기능에 사용하지 않는다.
- 화면 방향은 세로(portrait)로 고정한다. 가로 회전을 허용하지 않는다.

### NFR-007 퍼즐 수량 및 소진 처리

- 난이도별 퍼즐을 최소 20개 이상 사전 준비한다.
- 모든 퍼즐을 플레이한 경우 랜덤 반복 출제한다.

### NFR-003 성능

- 앱 시작 후 홈 화면 진입까지 3초 이내.
- 퍼즐 로드(난이도 선택 후 게임 화면 표시)까지 1초 이내.
- 숫자 입력 후 오류 검증 및 화면 갱신까지 200ms 이내.

### NFR-004 오프라인 동작

- 인터넷 연결 없이 모든 기능이 동작해야 한다.
- 퍼즐 데이터, 이미지, 폰트 등 모든 리소스는 앱 번들에 포함된다.

### NFR-005 확장 가능한 아키텍처

- 퍼즐 데이터는 데이터 레이어로 분리하여 추후 서버 API 교체가 가능한 구조로 구현한다.
- 난이도 추가 시 데이터 파일 추가만으로 대응 가능해야 한다.
- 문자열 리소스는 다국어 지원 구조로 분리한다.

### NFR-006 플랫폼 호환성

- Flutter를 사용하여 구현하며, 현재 단계는 Android만 지원한다.
- Android 최소 지원 버전: Android 8.0 (API 26) 이상.
- iOS는 Play Store 출시 준비 단계에서 별도 대응한다.

## 범위 외 항목 (Out of Scope)

아래 항목은 현재 버전에서 구현하지 않는다. Play Store 출시 준비 단계에서 별도 검토한다.

| 항목 | 제외 이유 |
|------|----------|
| 회원가입 / 로그인 | 2명의 가족 사용자 대상으로 불필요 |
| 서버 / DB 연동 | 오프라인 전용으로 충분 |
| AdMob 광고 | 사용 경험 저해, Play Store 출시 시 검토 |
| 랭킹 / 통계 / 기록 | 현 단계에서 불필요 |
| 메모 기능 (후보 숫자 표기) | UX 복잡도 증가, 시니어 사용자에 불필요 |
| Play Store 심사 대응 | 현재 APK 직접 설치 단계 |
| 타이머 / 점수 시스템 | 경쟁 압박 제거 (편안한 사용 목적) |
| 힌트 기능 | 우선순위 낮음, 추후 고려 |
| 다크 모드 | 우선순위 낮음, 추후 고려 |

| 항목 | 권장 기술 | 근거 |
|------|----------|------|
| 프레임워크 | Flutter (Dart) | Android/iOS 동시 대응, 단일 코드베이스 |
